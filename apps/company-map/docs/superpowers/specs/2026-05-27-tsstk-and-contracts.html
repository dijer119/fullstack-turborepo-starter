자사주(지분 통합) + 계약 카테고리

Date: 2026-05-27

Scope: ① pblntf_ty=B(주요사항보고) 중 자기주식 관련 공시를 지분 카테고리에 통합. ② "단일판매·공급계약체결" 등 계약 공시를 새 카테고리 "계약"으로 추가.

Goals

  • 자사주 취득/처분/소각 결정 → 지분 카테고리에 노출 (제목 + DART 링크 fallback).
  • 계약 공시 → 신규 "계약" 섹션 + ContractCard (계약일·기간·내용·상대방·지역·금액·매출액 대비).
  • SectionNav에 "계약" 추가, 4 섹션 navigation.

Non-Goals

  • 자사주 상세 API 호출 (취득결정·처분결정·신탁 등 4-5개 endpoint 복잡) — 후속.
  • 모든 주요사항보고(합병·분할·증자 등) 분류 — 후속.
  • 계약 외 영업 공시 (공급·취득 등) 세분화 — 후속.

Data Model

DisclosureCategory 확장

export type DisclosureCategory = "실적" | "배당" | "지분" | "계약";

ContractPayload

export interface ContractPayload {
  contractDate: string | null;       // 계약일 "YYYY-MM-DD"
  startDate: string | null;          // 계약기간 시작
  endDate: string | null;            // 계약기간 종료
  contractContent: string | null;    // 계약 내용 (예: "Valve remote control system외 판매계약")
  counterparty: string | null;       // 계약상대방
  supplyRegion: string | null;       // 공급지역
  amount: number | null;             // 계약금액 (원)
  recentSalesRatio: number | null;   // 최근매출액대비 %
}

OwnershipPayload — 변경 없음

자사주는 기존 reportType: "자기주식", detail: null로 저장. 카드는 기존 OwnershipCard의 자기주식 분기(제목+링크) 그대로.

DART API

list.json (pblntf_ty=B) — 자사주 + 계약 공통 호출

fetchOwnership·fetchContracts 각각 별도 호출하지 않고, B는 한 번 호출해서 두 함수가 공유. 또는 둘 다 호출하되 캐싱.

구현 단순화 우선: fetchOwnership에서 B 호출 → 자기주식 row 추출 (지분 카테고리로 저장). fetchContracts에서 B 호출 → 계약 row 추출 (계약 카테고리로 저장). 같은 corpCode·기간이라 DART 응답 동일. 코드 중복 허용 (DRY 우선보다 단순함).

자기주식 row 매칭

report_nm에 "자기주식" 포함 또는 "자사주" 포함. classifyOwnership의 기존 자기주식 매칭 그대로 사용.

계약 row 매칭

report_nm에 "단일판매" OR "공급계약" OR "판매계약체결" 포함.

단일판매·공급계약 상세 API

OpenDART 추정 endpoint: https://opendart.fss.or.kr/api/cnctSlsCnclSDcsn.json (Phase 2 alotMatter 사례처럼 추정. 정확한 응답은 구현 시 확인 단계 포함).

추정 응답 필드:

  • ctr_prd_bgn_dd, ctr_prd_end_dd: 계약기간 시작·종료
  • ctr_cn / cn: 계약 내용
  • cnt_pa_nm / ctr_pa_nm: 상대방
  • spy_aro_nm / regn: 공급지역
  • spy_amt / ctr_amt: 계약금액
  • rce_sl_amt_rat: 최근매출액 대비 %
  • ctr_dd: 계약일

응답 필드명 빗나갈 가능성 → Task에 응답 확인 단계 포함.

fetch 흐름

fetchDisclosuresForStock(code):
  try fetchEarnings
  try fetchOwnership   // 변경: D + B(자기주식 필터) 합쳐서 지분 카테고리에 저장
  try fetchDividends
  try fetchContracts   // 신규: B(계약 필터) → ContractPayload → 계약 카테고리

검증 기준

  • tsc EXIT=0. 마이그레이션 불필요.
  • "공시 갱신" → 종목당 추가 ≈ 1~2초 (계약 상세 row × N).
  • 005930 지분 카테고리에 자기주식 row 노출 (취득결정/처분결정/소각 등).
  • 005930 또는 계약 공시가 있는 종목 페이지에 계약 카드 노출, 계약일/기간/금액/상대방 채워짐.
  • SectionNav 4개 항목 노출, 계약 섹션으로 스크롤 동작.
  • 계약 상세 API 실패 시 ContractPayload 모든 필드 null → 카드는 제목+공시일+DART 링크만.

Out of Scope / Open Questions

  • 계약 상세 API endpoint와 응답 필드명은 구현 단계에서 실제 응답 검증 (Phase 2 alotMatter·elestock 패턴).
  • 자사주 상세 (취득·처분·소각 결정 별 결정 사유·수량 등) — 별도 Phase.
  • 계약이 너무 많은 종목 (예: SI 회사)은 max 16건 제한으로 잘림.
