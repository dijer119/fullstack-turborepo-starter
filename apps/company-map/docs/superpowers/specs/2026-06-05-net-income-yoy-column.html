순이익(당기순이익) YoY 컬럼 추가 — Design Spec

Date: 2026-06-05

Scope: /stocks(전종목 조회) 리스트에 당기순이익 전년동기 대비(YoY) 컬럼을 추가한다. 기존 영업이익 YoY와 동일한 파이프라인을 재사용하며, 데이터는 영업이익을 가져오는 동일한 DART fnlttSinglAcnt 응답에서 추출하므로 추가 DART 호출이 없다.

Goals

  • 리스트에서 영업이익 YoY 옆에 순이익 YoY를 함께 확인 — 수익성 변화를 영업·순이익 두 관점으로 비교.
  • 기존 영업이익 YoY 수집 잡(operating_income)에 순이익 추출을 얹어 DART 호출 추가 없이 채운다.
  • 두 YoY 컬럼을 라벨로 명확히 구분(영업이익 YoY / 순이익 YoY).

Non-Goals

  • 순이익 QoQ(직전 보고 대비) — YAGNI. 리스트는 YoY만 표시하므로 prevReport/history 저장 안 함.
  • 순이익 기준 정렬 옵션 추가 — 요청 외. 정렬은 현행(YoY 정렬=영업이익) 유지.
  • 지배주주지분 순이익 등 별도 분해 — 영업이익과 동일하게 연결 당기순이익(손실) 총액만 사용.

Decisions

항목결정
데이터 소스영업이익과 동일한 fnlttSinglAcnt 응답 (추가 호출 0)
순이익 계정account_nm ∈ {"당기순이익","당기순이익(손실)"}, 연결재무제표(CFS) 우선
YoY 산식당기(thstrm) vs 전년동기(frmtrm) — 영업이익과 동일
저장FinancialSnapshot에 net income 3필드 추가(prevReport 제외)
컬럼 배치기존 "YoY"→"영업이익 YoY"로 변경, 옆에 "순이익 YoY" 추가
갱신기존 수동 잡 operating_income 그대로 (자동 아님)

데이터 모델 (Prisma · SQLite)

FinancialSnapshot에 필드 3개 추가. 마이그레이션 후 prisma generate 필요. (DATABASE_URL 오버라이드 금지 — 실제 DB data/company-map.db 사용.)

model FinancialSnapshot {
  code               String   @id
  latestBsnsYear     Int      @map("latest_bsns_year")
  latestReprtCode    String   @map("latest_reprt_code")
  opIncome           BigInt?  @map("op_income")
  opIncomeYoyBase    BigInt?  @map("op_income_yoy_base")
  opIncomePrevReport BigInt?  @map("op_income_prev_report")
  opIncomeYoyPct     Float?   @map("op_income_yoy_pct")
  netIncome          BigInt?  @map("net_income")           // 추가: 당기 순이익(thstrm)
  netIncomeYoyBase   BigInt?  @map("net_income_yoy_base")  // 추가: 전년동기(frmtrm)
  netIncomeYoyPct    Float?   @map("net_income_yoy_pct")   // 추가: 정렬/참고용 계산값
  fetchedAt          DateTime @default(now()) @map("fetched_at")

  master  StockMaster @relation(fields: [code], references: [code], onDelete: Cascade)

  @@map("financial_snapshots")
}

추출 함수 (src/lib/dart/operating-income.ts)

extractOpIncome와 동일 패턴의 extractNetIncome(response) 추가. 실제 DART 응답에서 순이익 계정명은 "당기순이익(손실)"로 들어오며 연결/별도 두 행이 모두 존재 → 연결(CFS) 우선, 첫 유효 행의 {thstrm, frmtrm} 반환.

수집 (scripts/refresh-operating-income.ts)

  • 기존 루프에서 이미 가져온 resp로 extractNetIncome(resp)도 호출 → collected 항목에 순이익 {thstrm, frmtrm} 동봉.
  • 최신 보고서 선택 로직(연도·보고서 우선순위)은 그대로 재사용. 선택된 최신 항목의 순이익으로 FinancialSnapshot의 netIncome/netIncomeYoyBase/netIncomeYoyPct를 영업이익과 함께 upsert.
  • netIncomeYoyPct = frmtrm≠0이면 (thstrm-frmtrm)/|frmtrm|×100, 아니면 null (영업이익과 동일 산식).
  • DART 호출 추가 없음 — 같은 응답 재사용.
백필: 기존 스냅샷에는 순이익 필드가 비어 있으므로, 이 수동 잡(operating_income = "데이터 업데이트")을 한 번 재실행해야 채워진다. 그 전까지 순이익 YoY는 "—"로 표시.

액션 (src/actions/stocks-explorer.ts)

  • StocksExplorerRow에 netIncome: number | null, netIncomeYoyBase: number | null 추가 (영업이익 필드 옆).
  • financialSnapshot에서 BigInt → Number 변환해 매핑 (opIncome과 동일 방식).

UI (src/app/stocks/StocksExplorerClient.tsx)

  • 기존 "YoY" 헤더 → "영업이익 YoY"로 라벨 변경.
  • 그 옆에 "순이익 YoY" 헤더·셀 추가.
  • 셀 값 = computeGrowth(r.netIncome, r.netIncomeYoyBase), 표시·색·툴팁은 기존 formatGrowth/growthColorClass/titleForGrowth 재사용.
  • 확장 행 colSpan 등 컬럼 수에 의존하는 값이 있으면 +1 보정.

테스트 (Vitest)

  • extractNetIncome 단위 테스트 — 실제 DART fnlttSinglAcnt 응답(삼성전자 등) 픽스처에서 연결 당기순이익(손실)의 thstrm/frmtrm를 정확히 뽑고, 법인세차감전 순이익/총포괄손익은 제외하는지 확인.
  • 기존 extractOpIncome 테스트가 있으면 회귀 확인.
