종목별 공시 History — Phase 1 (실적)

Date: 2026-05-26

Scope: 종목별 공시 timeline 페이지 신규 구축. Phase 1은 "실적" 카테고리(정기보고서)만 완전 구현. 다른 카테고리는 후속 Phase.

Phase Scope

Phase내용본 spec
Phase 1페이지 + 라우팅 + 실적 카테고리 카드✅ 본 문서
Phase 2지분 변동(VIP 통합) + 기타 raw 카테고리별도 spec
Phase 3M&A · 배당 · 감사 등 세분화, 카드 디자인 정교화별도 spec

Goals

  • 종목 단위 페이지 /stocks/[code] 신규 — 진입은 종목명 옆 📋 아이콘.
  • 실적 timeline 카드: 보고서명·시기·매출/영업익/순이익 + YoY/QoQ + 공시 일시.
  • DART API 호출 결과 DB 캐시(Disclosure 테이블), lazy fetch + 명시적 refresh.

Non-Goals (Phase 1)

  • 지분/감사/배당 등 다른 카테고리 카드 — Phase 2/3로 이연.
  • 애널리스트 컨센서스 "예상대비" 표시 — 데이터 소스 없음.
  • 모든 종목 사전 백그라운드 batch fetch — lazy 방식만.
  • 공시 검색/필터 UI — Phase 후속.

Data Model

신규 테이블 Disclosure. StockMaster와 N:1.

model Disclosure {
  rcpNo      String   @id @map("rcp_no")          // DART 접수번호 (unique)
  code       String                                // 종목 코드
  corpCode   String   @map("corp_code")
  reportNm   String   @map("report_nm")
  pblntfTy   String   @map("pblntf_ty")            // DART 대분류 (A/B/C/D/...)
  rceptDt    DateTime @map("rcept_dt")             // 공시일
  category   String                                // 우리 분류: "실적" | "기타" (Phase 1은 두 값만)
  payload    String?                               // JSON 직렬화. 실적은 EarningsPayload.
  fetchedAt  DateTime @default(now()) @map("fetched_at")

  master     StockMaster @relation(fields: [code], references: [code], onDelete: Cascade)

  @@index([code, rceptDt])
  @@index([category])
  @@map("disclosures")
}

StockMaster에 disclosures Disclosure[] 관계 추가.

EarningsPayload (JSON, 실적 카드 전용)

interface EarningsPayload {
  bsnsYear: number;         // 사업연도
  reprtCode: "11013" | "11012" | "11014" | "11011";  // 1Q / 반기 / 3Q / 사업
  reprtLabel: string;       // "26Q1" | "26반기" | "26Q3" | "26사업"
  consolidated: boolean;    // 연결/별도. CFS=연결, OFS=별도
  // 단위: 원
  revenue: number | null;       // 매출
  opIncome: number | null;      // 영업이익
  netIncome: number | null;     // 순이익
  // YoY/QoQ는 클라이언트 계산 또는 payload에 포함 — 본 spec에서는 payload에 포함.
  revenueYoyPct: number | null;
  opIncomeYoyPct: number | null;
  netIncomeYoyPct: number | null;
  revenueQoqPct: number | null;
  opIncomeQoqPct: number | null;
  netIncomeQoqPct: number | null;
}

DART API 흐름

list.json (공시 목록)

엔드포인트: https://opendart.fss.or.kr/api/list.json

파라미터:

  • corp_code — 8자리 DART 회사 코드 (이미 StockMaster.corpCode에 있음).
  • pblntf_ty=A — 정기공시만 (Phase 1).
  • bgn_de=20180101 — 충분히 과거부터.
  • end_de — 오늘.
  • page_count=100, 페이지네이션은 page_no.

fnlttSinglAcnt.json (재무 지표) — 보고서별 N회 호출

이미 src/lib/dart/financial.ts의 fetchDartFinancial(corpCode, year, reprtCode) 존재. 재사용.

응답에서 추출:

  • 매출액: account_nm = "매출액" 또는 "수익(매출액)" (금융업 차이) → thstrm_amount.
  • 영업이익: "영업이익" 또는 "영업이익(손실)" → 이미 extractOpIncome 패턴 적용.
  • 당기순이익: "당기순이익" 또는 "당기순이익(손실)".
  • fs_div: CFS(연결) 또는 OFS(별도). 우선순위: CFS > OFS.

YoY: (thstrm - frmtrm) / |frmtrm| × 100. QoQ는 직전 분기 데이터 비교 — 같은 사업연도의 직전 reprtCode 결과를 활용하거나 별도 호출.

Fetch 패턴

  1. 페이지 진입 → server component가 db.disclosure.findMany({ where: { code, category: "실적" }, orderBy: { rceptDt: "desc" } }) 호출.
  2. 결과 0건이면 → fetchDisclosuresForStock(code) server action 호출 (lazy seed).
    • list.json에서 정기보고서 목록 가져옴.
    • 각 row마다 fnlttSinglAcnt.json 호출해서 매출/영업익/순이익 추출, payload 만들어 upsert.
    • API 부하 고려: 한 종목당 최대 N건(예: 최근 4년 × 4분기 = 16건) 제한.
  3. 페이지에 "공시 갱신" 버튼 → 강제 재실행.
  4. 에러 발생 시 console.error + 빈 상태 표시. 사용자가 refresh로 재시도.

진입 — 종목명 옆 아이콘

StocksExplorerClient.tsx의 종목명 셀에 작은 📋 (FileText 또는 ExternalLink 류) 아이콘 추가. 클릭 시 /stocks/{code}로 이동. 종목명 텍스트는 기존 네이버 금융 외부링크 유지(기존 동작 보존).

표시 규칙

  • 금액 포맷: 1억 미만 원 표기, 1억 이상 "X억", 1조 이상 "X.X조".
  • YoY/QoQ: 부호 포함 (예: +5%, -12%). 색상: 양수=녹색, 음수=빨강, 0%=회색.
  • NULL 값: —.
  • 공시 일시: YYYY.MM.DD HH:mm (rceptDt 포맷).
  • 카드는 rceptDt desc 순 (최신 위).

SQL:

CREATE TABLE disclosures (
  rcp_no TEXT PRIMARY KEY,
  code TEXT NOT NULL,
  corp_code TEXT NOT NULL,
  report_nm TEXT NOT NULL,
  pblntf_ty TEXT NOT NULL,
  rcept_dt DATETIME NOT NULL,
  category TEXT NOT NULL,
  payload TEXT,
  fetched_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (code) REFERENCES stock_masters(code) ON DELETE CASCADE
);
CREATE INDEX idx_disclosures_code_rcept_dt ON disclosures(code, rcept_dt);
CREATE INDEX idx_disclosures_category ON disclosures(category);

검증 기준

  • 마이그레이션 성공, disclosures 테이블 + 인덱스 존재.
  • tsc EXIT=0.
  • /stocks 종목명 옆 📋 아이콘 클릭 → /stocks/005930 이동, 200 응답.
  • 첫 진입 시 자동 fetch + 카드 표시 (DART API 응답 대기 시간 ≈ 수 초).
  • "공시 갱신" 버튼 → 같은 종목 다시 fetch, payload 최신화.
  • 삼성전자(005930) 페이지에 최근 4년 정기보고서 4~16건 카드 노출, 각 카드는 매출/영업익/순이익 + YoY/QoQ.
  • 은행(KB금융 등)도 매출 "수익(매출액)" 매핑 정상 동작.
  • 존재하지 않는 코드 /stocks/000000 → 404 또는 빈 상태.

Out of Scope / Open Questions

  • 분기보고서 외 정기공시(사업/반기) YoY/QoQ 계산은 reprtCode 매칭 로직이 필요. 본 Phase에서는 분기끼리·반기끼리만 비교.
  • 금융업(은행/보험)의 매출 개념이 일반 산업과 다름 — Phase 1에서는 "수익(매출액)" 표시만, 해석 주의 표기 없음.
  • 최초 fetch가 느릴 경우 UX (로딩 표시·취소 등) — 본 Phase는 단순 "로딩 중…" 텍스트.
