액티브 ETF 구성종목·비중 변화 추적 — Design Spec

Date: 2026-06-03

Scope: /stocks(전종목 조회) 하위에 /stocks/etf 페이지 신설. 관심 ETF를 코드로 등록하면 worker가 매 영업일 KRX PDF(구성종목·비중)를 스냅샷으로 저장하고, ETF 선택 시 최신 스냅샷을 직전 스냅샷과 비교해 비중 상위 10종목의 신규 진입·이탈·비중 증감을 보여준다.

Goals

  • 관심 ETF(예: 액티브 ETF 0074K0)의 구성종목과 비중을 한눈에 확인.
  • 직전 영업일 대비 구성종목·비중 변화(신규 편입 / 이탈 / 비중 증감 %p)를 시각적으로 표시.
  • worker 자동 일별 수집으로 변화 이력이 별도 조작 없이 누적.

Non-Goals

  • 액티브 ETF 전체/전체 ETF 자동 추적 — YAGNI. 사용자가 등록한 관심 ETF만 다룬다.
  • 임의 두 날짜 비교, 기간(N일) 비교 — 직전 스냅샷 대비만. (추후 확장 여지)
  • 비중 상위 10위 밖 종목 저장 — 스냅샷당 상위 10종목만 저장.
  • Naver 페이지 스크래핑 — 데이터 소스는 KRX MDC.

Decisions (확정된 요구사항)

항목결정

데이터 모델 (Prisma · SQLite)

company-map 전용 SQLite + 별도 Prisma 클라이언트(@prisma-clients/company-map)에 테이블 3개 추가. 마이그레이션 후 prisma generate 필요.

model EtfWatch {
  code      String   @id            // KRX 단축코드 (예: "0074K0")
  isin      String?  @map("isin")   // KRX 표준코드/ISIN (등록 시 1회 해석)
  name      String
  createdAt DateTime @default(now()) @map("created_at")

  snapshots EtfPdfSnapshot[]
  @@map("etf_watches")
}

model EtfPdfSnapshot {
  id        String   @id @default(uuid())
  etfCode   String   @map("etf_code")
  trdDd     String   @map("trd_dd")     // 기준일 yyyymmdd
  fetchedAt DateTime @default(now()) @map("fetched_at")

  watch     EtfWatch       @relation(fields: [etfCode], references: [code], onDelete: Cascade)
  holdings  EtfPdfHolding[]

  @@unique([etfCode, trdDd])
  @@index([etfCode, trdDd])
  @@map("etf_pdf_snapshots")
}

model EtfPdfHolding {
  id              String  @id @default(uuid())
  snapshotId      String  @map("snapshot_id")
  constituentCode String  @map("constituent_code")
  constituentName String  @map("constituent_name")
  weight          Float?                              // 비중 %
  shares          Float?                              // 주식수/계약수
  amount          Float?                              // 평가금액(원)

  snapshot        EtfPdfSnapshot @relation(fields: [snapshotId], references: [id], onDelete: Cascade)
  @@index([snapshotId])
  @@map("etf_pdf_holdings")
}

스냅샷당 EtfPdfHolding은 비중 상위 10행으로 제한한다.

데이터 파이프라인

KRX 연동 (worker/fetch-krx-etf.ts)

기존 worker/fetch-krx.ts의 헤더·세션·lastBusinessDay() 패턴을 재사용한다.

  • resolveEtfIsin(code) — ETF 전종목 목록(MDCSTAT04601 계열)에서 단축코드 → ISIN 해석. ETF 등록 시 1회 호출.
  • fetchEtfPdf(isin, trdDd) — 해당 ETF의 PDF 구성종목(MDCSTAT05001 계열) 호출 → 종목코드·명·비중·주식수·평가금액 행 파싱.
⚠️ KRX 엔드포인트 검증 한계 / OPEN ITEM: 설계 세션의 네트워크에서는 KRX가 IP를 차단(LOGOUT 400)하여 worker가 쓰는 기존 엔드포인트(MDCSTAT01501)조차 직접 호출하지 못했다. 이는 엔드포인트 문제가 아니라 환경 차단이며, company-map worker는 실제 런타임에서 KRX를 정상 호출 중이다. 따라서 새 ETF 목록/PDF 엔드포인트의 정확한 bld 값과 파라미터, 응답 필드명은 구현 단계에서 앱 런타임(worker 또는 일회성 tsx 스크립트)으로 확정하고, 실제 응답을 픽스처로 저장해 파서 단위 테스트를 작성한다.

스냅샷 수집 (worker/etf-pdf-loop.ts)

  • 각 EtfWatch에 대해 최신 영업일 PDF를 가져와 (etfCode, trdDd) upsert.
  • 해당 기준일 스냅샷이 이미 있으면 skip — 2분 worker 루프가 같은 날 중복 수집하지 않게 한다.
  • 저장 시 비중 내림차순 정렬 후 상위 10종목만 EtfPdfHolding으로 기록.
  • worker 메인 루프(worker/index.ts)에 추가. 관심 ETF가 소수라 부하가 가벼워 NCAV 배치와 달리 자동 루프에 포함 가능.

수동 갱신

  • refresh-jobs에 kind: "etf_pdf" 추가 → /stocks/etf의 "구성종목 갱신" 버튼에서 즉시 실행.

변화 계산 (순수 함수)

diffHoldings(latest, prev) — 최신·직전 스냅샷의 상위 10 holdings를 비교해 종목별 변화 산출. UI·테스트에서 공용.

interface HoldingChange {
  constituentCode: string;
  constituentName: string;
  weight: number | null;          // 최신 비중
  weightDelta: number | null;     // %p (최신 - 직전), 신규면 null/+전체
  shares: number | null;
  sharesDelta: number | null;
  status: "신규" | "유지" | "이탈"; // Top10 기준
}
  • 비중 내림차순 정렬. 이탈 종목(직전 Top10에 있었으나 최신 Top10에 없음)은 테이블 하단에 별도 표기.
  • 라벨 의미: 상위 10종목만 저장하므로 "신규/이탈"은 ETF 전체 편입·편출이 아니라 비중 상위 10위 진입·이탈을 뜻한다. UI 라벨은 신규(Top10 진입) / 이탈(Top10 밖)로 명시한다.

UI (/stocks/etf)

  • 등록 영역: 코드 입력으로 ETF 추가 / 삭제. 추가 시 ISIN 해석 + 즉시 1회 스냅샷 수집.
  • ETF 선택 → 구성종목 변화 테이블: 컬럼 = 종목명 · 비중 · 비중Δ(%p, 증가 녹색/감소 빨강) · 상태(신규🟢/이탈🔴) · 주식수 · 주식수Δ. 상단에 기준일 vs 직전일 표기.
  • "구성종목 갱신" 버튼 (수동 트리거, 기존 RefreshButton/RefreshMenu 패턴).
  • 전종목 조회 페이지·메뉴(AppMenu/ActiveNav)에서 진입 링크 추가.

에러 처리 / 엣지 케이스

  • KRX 호출 실패: 스냅샷 미생성(기존 데이터 유지), 화면에 경고 표시. 개별 ETF 실패가 다른 ETF 수집을 막지 않는다.
  • 스냅샷이 1개뿐(직전 없음): 변화 없이 현재 구성만 표시("비교할 직전 데이터 없음").
  • 신규 ETF 등록 직후: 당일 스냅샷만 존재 → 위와 동일 처리.
  • 단축코드 → ISIN 해석 실패: 등록 거부 + 에러 메시지.

테스트 (Vitest)

  • diffHoldings 순수 함수: 신규 진입 / 이탈 / 비중 증감 / 주식수 증감 케이스.
  • KRX PDF 파서: 실제 응답 픽스처 기반 파싱 테스트(구현 시 앱 런타임에서 캡처).
  • 상위 10 제한 로직: 11개 이상 입력 시 비중 상위 10만 저장되는지.
