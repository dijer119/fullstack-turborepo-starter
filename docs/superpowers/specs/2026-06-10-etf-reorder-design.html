관심 ETF 노출 순서 변경 (드래그 앤 드롭) — 설계 명세

작성일: 2026-06-10 · 대상 앱: apps/company-map · 페이지: /stocks/etf

1. 목적

/stocks/etf 상단의 관심 ETF 버튼 나열 순서를 사용자가 드래그 앤 드롭으로 바꾸고, 그 순서가 DB에 저장되어 새로고침 후에도 유지되게 한다. 현재는 createdAt 등록순 고정이다.

2. 결정 사항

항목결정근거
순서 변경 UI드래그 앤 드롭사용자 선택
구현 방식네이티브 HTML5 DnD (draggable + onDragStart/Over/Drop)의존성 추가 0. 데스크탑 전용으로 충분 (사용자 확인)

순서 저장EtfWatch.sortOrder Int 컬럼SQLite 단일 파일 DB, 영속 필요

3. 스키마 변경

model EtfWatch {
  code      String   @id
  isin      String?  @map("isin")
  name      String
  sortOrder Int      @default(0) @map("sort_order")   // ← 추가
  createdAt DateTime @default(now()) @map("created_at")
  ...
}
  • 마이그레이션: yarn workspace company-map prisma migrate dev --name etf_watch_sort_order
  • 백필 불필요: 조회를 [{ sortOrder: "asc" }, { createdAt: "asc" }] 이중 정렬로 하면 기존 행(전부 0)은 종전 등록순이 그대로 유지된다. 첫 드래그 시 전체 행에 0..N-1이 부여되며 그때부터 sortOrder가 단독으로 의미를 가진다.
  • 신규 등록: max(sortOrder) + 1로 맨 뒤에 추가 (빈 테이블이면 0).
⚠️ 주의: migrate/generate 후 실행 중인 dev 서버는 옛 PrismaClient 싱글턴을 유지하므로 반드시 dev 서버를 재시작해야 한다 (compound learning 기록 사항).

4. 서버 변경 (src/actions/etf.ts)

함수변경
listEtfWatches()orderBy만 [{ sortOrder: "asc" }, { createdAt: "asc" }]로 교체. 반환 타입 변화 없음.
registerEtf()생성 전 aggregate({ _max: { sortOrder } })로 최댓값 조회, (max ?? -1) + 1을 sortOrder로 저장.
reorderEtfWatches(codes: string[]) (신규)검증: 입력 codes가 등록된 전체 코드 집합과 정확히 일치(개수·구성)해야 함 — 불일치 시 { ok: false }. 통과 시 db.$transaction(codes.map((code, i) => update sortOrder = i)) 후 revalidatePath("/stocks/etf"), { ok: true }.

5. 재배열 순수 함수 (src/lib/etf/reorder.ts 신규)

// drag 코드를 drop 코드 위치에 삽입한 새 배열. 입력 불변.
export function moveCode(codes: string[], dragCode: string, dropCode: string): string[];
  • 동작: 표준 arrayMove 의미론 — dragCode를 원래 위치에서 제거하고 dropCode의 원래 인덱스 위치에 삽입한다. 결과적으로 드래그 항목이 드롭 대상의 자리를 차지하며, 앞→뒤 이동 시 대상 바로 뒤, 뒤→앞 이동 시 대상 바로 앞에 놓인다. 인접 항목에 드롭하면 양방향 모두 교환되고, 마지막 항목에 드롭하면 맨 끝으로 이동할 수 있다. (대안이었던 "드롭 대상 앞 삽입" 방식은 인접 항목의 오른쪽 이동이 no-op이 되고 맨 끝 이동이 불가능해 배제.)
  • 엣지: dragCode === dropCode이거나 둘 중 하나가 배열에 없으면 원본과 같은 내용의 새 배열 반환 (변경 없음).
  • Vitest 테스트 케이스: 앞→뒤 이동, 뒤→앞 이동, 인접 항목 교환(양방향), 맨 끝 이동, 자기 자신에 드롭, 미존재 코드, 입력 배열 불변성.

6. UI 변경 (src/app/stocks/etf/EtfManager.tsx)

  • ETF 버튼 wrapper <span>에 draggable + onDragStart(드래그 코드 state 저장) / onDragOver(preventDefault() + 오버 코드 state) / onDrop / onDragEnd(state 초기화) 부착.
  • 낙관적 업데이트: 컴포넌트가 ETF 순서를 로컬 state(orderedCodes)로 관리. 드롭 시 moveCode() 결과를 즉시 state에 반영 → reorderEtfWatches() 호출 → 실패(ok: false 또는 throw) 시 router.refresh()로 서버 상태 복원. 성공 시에도 router.refresh()로 서버와 동기화.
  • 로컬 state는 watches prop에서 파생하되, prop 변경(서버 refresh) 시 동기화한다. 렌더 시 orderedCodes 순서로 watches를 정렬해 표시 (코드→watch 맵 사용, orderedCodes에 없는 신규 코드는 뒤에 표시).
  • 시각 피드백: 드래그 중인 버튼은 opacity-50, 드래그 오버된 버튼 wrapper에는 왼쪽 삽입 인디케이터 (border-l-2 border-blue-500).
  • 드래그 시작은 ETF 버튼 영역에서만 — × 삭제 버튼 클릭과 충돌하지 않게 wrapper 단위 draggable이면 충분 (× 클릭은 click 이벤트라 드래그와 간섭 없음).

7. 에러 처리

  • reorderEtfWatches: 코드 집합 불일치(동시 추가/삭제로 어긋난 경우) → { ok: false }, 클라이언트는 router.refresh()로 최신 상태 복원.
  • DB 오류 throw → 클라이언트 catch에서 동일하게 router.refresh().
  • moveCode는 어떤 입력에도 throw하지 않는다.
