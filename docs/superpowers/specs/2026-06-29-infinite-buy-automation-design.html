무한매수법 자동매매 (토스증권 연동) 설계

날짜: 2026-06-29 · 앱: company-map · 출처: 라오어 「미국주식 무한매수법」 (블로그 리뷰 jykang73/224324707916)

1. 목표

미국 3배 레버리지 ETF(TQQQ 등)를 대상으로 라오어 무한매수법의 핵심 사이클을 토스증권 Open API로 완전 무인 자동 실행하고, 종목별 진행 현황을 대시보드로 본다.

성공 기준:

  • active 사이클에 대해 worker가 US 거래일마다 1회 자동으로 그날의 LOC 매수·목표 매도 주문을 제출한다(또는 dryRun이면 로깅).
  • 대시보드에서 종목별 진행 회차·라이브 평단·보유·평가수익률·오늘 주문·다음 매도가를 확인한다.
  • 전략 코어가 순수 함수로 분리되어 Vitest로 검증된다.
실자금 자동매매 경고. 매수/매도는 실제 라이브 계좌에서 체결된다. 본 기능은 사이클별 dryRun 기본 ON으로 시작하며, 모의 주문 로그 확인 + CLS(LOC) 소액 실거래 1회 검증 후 사용자가 LIVE로 전환한다. 전역 킬스위치로 전체 중단 가능.

2. 무한매수법 핵심 사이클 (구현 범위 = 상황1)

요소규칙
자금 분할dailyAmount = 원금 / 분할수 (기본 분할수 40)
1회차장중 MARKET 매수 1회 분량으로 사이클 시작
2회차~dailyAmount 절반 → BUY LIMIT CLS @ 평단(LOC 평단), 절반 → BUY LIMIT CLS @ 현재가×(1+큰수%)(LOC 큰수). 각 ≥1주
매일 매도보유 전량 SELL LIMIT DAY @ 평단×(1+익절%) (기본 +10%). 당일 만료 → 매일 재등록
40회차 소진 후매수 중단, 매도 목표가만 유지. 평가수익률 ≤ -10%면 MARKET 전량 손절. 익절 체결(보유 0) 또는 손절 시 → round 0으로 리셋(재시작)

제외(범위 밖): 상황2(원금 소진 후 -10% 초과 폭락 시 동결 + 새 계좌 + 손익통산). 다계좌·복잡도 큼 → 후속 작업.

LOC 구현: 토스 createOrder의 timeInForce: "CLS" + orderType: "LIMIT" = Limit On Close. (LOC 매수=지정가보다 종가 낮으면 체결 / LOC 매도=지정가보다 종가 높으면 체결.)

3. 데이터 모델 (company-map SQLite, 신규)

model InfiniteBuyCycle {
  id            String   @id @default(uuid())
  symbol        String                        // 토스 심볼 (US 티커, 예: TQQQ)
  name          String                        // 표시용 종목명
  accountSeq    Int?                          // 토스 계좌(null=기본계좌)
  principal     Float                         // 원금 (USD)
  splits        Int      @default(40)         // 분할 회차 수
  profitTarget  Float    @default(10)         // 익절 % (평단 대비)
  bigBuyPremium Float    @default(12)         // "큰수" LOC 매수 프리미엄 %
  lossCut       Float    @default(10)         // 소진 후 손절 기준 % (수익률 ≤ -lossCut)
  status        String   @default("active")   // active | paused | completed | stopped
  dryRun        Boolean  @default(true)       // 모의(주문 미제출) 기본 ON
  round         Int      @default(0)          // 현재까지 매수 완료 회차
  lastRunDate   String?                       // 마지막 실행 US 거래일 YYYYMMDD (멱등)
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt
  orders        InfiniteBuyOrder[]
  @@map("infinite_buy_cycles")
}

model InfiniteBuyOrder {
  id          String   @id @default(uuid())
  cycleId     String
  tradeDate   String                          // US 거래일 YYYYMMDD
  round       Int
  side        String                          // BUY | SELL
  kind        String                          // loc_avg | loc_big | target_sell | first_market | reset_sell
  orderType   String                          // LIMIT | MARKET
  tif         String                          // CLS | DAY
  price       Float?
  quantity    Float
  tossOrderId String?                         // 실제 주문 ID (dryRun이면 null)
  dryRun      Boolean
  status      String                          // submitted | failed | simulated
  error       String?
  createdAt   DateTime @default(now())
  cycle       InfiniteBuyCycle @relation(fields: [cycleId], references: [id], onDelete: Cascade)
  @@index([cycleId, tradeDate])
  @@map("infinite_buy_orders")
}

평단·보유수량·평가수익률은 저장하지 않고 토스 보유종목 API에서 라이브 조회한다(사이클은 설정 + 회차 + 주문이력만 보관).

4. 전략 코어 (순수 함수)

src/lib/infinite-buy/strategy.ts — computeDailyOrders(state). 토스 의존 없는 순수 계산이라 Vitest로 단위 검증.

interface CycleState {
  round: number; splits: number; principalUsd: number;
  profitTarget: number; bigBuyPremium: number; lossCut: number;
  avgPrice: number | null;     // 토스 라이브 평단 (보유 없으면 null)
  currentPrice: number;        // 토스 라이브 현재가
  holdingQty: number;          // 토스 라이브 보유수량
  pnlPct: number | null;       // 토스 라이브 평가수익률
}
interface IntendedOrder {
  side: "BUY"|"SELL"; kind: string; orderType: "LIMIT"|"MARKET";
  tif: "CLS"|"DAY"; price: number | null; quantity: number;
}
function computeDailyOrders(s: CycleState): {
  orders: IntendedOrder[];
  nextRound: number;           // 매수 발생 시 round+1, 아니면 동일
  resetAfter: boolean;         // 손절/익절완료로 리셋해야 하면 true
}

로직 분기:

  • round < splits (매수 단계):
    • round==0 → first_market BUY MARKET (qty = floor(dailyAmount/currentPrice), ≥1)
    • round≥1 → loc_avg BUY LIMIT CLS @ avgPrice (qty = floor((dailyAmount/2)/avgPrice), ≥1) + loc_big BUY LIMIT CLS @ currentPrice×(1+bigBuyPremium/100) (qty ≥1)
    • + target_sell SELL LIMIT DAY @ avgPrice×(1+profitTarget/100), qty = holdingQty (보유>0일 때)
    • nextRound = round+1
  • round ≥ splits (소진 후, 상황1):
    • holdingQty==0 → 익절 완료로 간주 → resetAfter=true (주문 없음)
    • pnlPct ≤ -lossCut → reset_sell SELL MARKET 전량 → resetAfter=true
    • 그 외 → target_sell SELL LIMIT DAY @ avgPrice×(1+profitTarget/100) 유지

금액·수량은 USD 기준. 매수 가능금액 부족 시 해당 주문 skip + 로깅(코어는 의도만 계산, 가용성 체크는 실행기에서).

5. Worker 루프 (완전 무인)

worker/infinite-buy-loop.ts (기존 etf-pdf-loop 패턴, worker/index.ts에 등록). 주기 폴링하며 US 정규장 세션 + 거래일당 1회 실행:

  1. getMarketStatuses(US 세션)·getMarketCalendarUS로 정규장/개장일 확인, 주말·휴장 skip.
  2. 각 status==active 사이클: lastRunDate != 오늘(US 거래일)이면 실행.
  3. 토스 라이브(보유·현재가·평단·수익률) 조회 → computeDailyOrders → 주문 제출(dryRun이면 로깅만, status:"simulated").
  4. 매수가능금액(getBuyingPower) 부족분 주문은 skip+로깅. round/lastRunDate/리셋 반영.

실행 코어는 src/lib/infinite-buy/run.ts의 runCycle(db, toss, cycle)로 분리해 worker·수동 트리거·테스트가 공유한다.

7. 안전장치 (실자금 필수)

  • dryRun 기본 ON (사이클별). LIVE 전환은 명시적 토글.
  • 전역 킬스위치: 환경변수(예: INFINITE_BUY_DISABLED=1) 또는 설정행 → 모든 자동 제출 차단(dryRun 강제).
  • 거래일당 멱등(lastRunDate) — 중복 주문 방지.
  • 원금·40회차 상한 준수, 매수가능금액 초과 주문 차단.
  • 모든 주문(모의 포함) InfiniteBuyOrder 로깅 + 에러 기록.
  • 고액주문(confirmHighValueOrder) 플래그 적절히 전달.

9. 검증 계획

  • 전략 코어: Vitest — 1회차 MARKET, 2회차 LOC 평단/큰수 수량·가격, 목표 매도가, 소진 후 손절/익절리셋, 경계(보유 0, 평단 null, 가용금액 부족).
  • 실행기: dryRun 모드로 runCycle 실행 → 주문 미제출 + InfiniteBuyOrder simulated 로깅 확인(라이브 토스 조회는 읽기 전용).
  • UI: HMR 반영, 대시보드 렌더·등록·토글 확인. (스키마 변경 후 dev 서버 재시작 필수.)
  • LIVE 검증: 사용자가 소액 1종목으로 dryRun→LIVE 전환 후 CLS 주문 1회 실제 제출·체결 확인.
  • lint: 변경 파일만 npx eslint <files> (워크스페이스 lint는 기존 에러로 게이트 불가).

10. 가정 / 범위 밖

  • 1종목 = 1 활성 사이클·전용 계좌/보유 — 평단·보유를 토스 라이브로 쓰므로 전략 밖 동일종목 보유 시 평단이 섞임.
  • CLS(LOC) 실동작 미검증 — 타입엔 존재. LIVE 전 소액 1회 실거래로 검증 필요.
  • 스케줄: US 정규장 게이트로 거래일당 1회. (DST 변동은 세션 상태로 흡수.)
  • 범위 밖: 상황2(동결/새 계좌/손익통산), 주문 정정·취소·체결 추적, 환전 자동화, 한국 종목.
  • company-map 주의: Prisma 스키마 변경 후 dev 서버 재시작 필수, dev 백그라운드 금지(turbo가 3004 관리).
