1. 목표 · 확정 결정

  • company-map 무한매수 자동매매에 V4.0 일반모드를 별도 버전("v4.0")으로 추가한다. 기존 v1·v2.2 경로는 무변경.
  • 범위 = 코어만 (사용자 확정): 별지점 매수(전반 ½+½ / 후반 전액) + 2단 매도 + 연속 T·잔금 영속화 + 체결 동기화 자동화. LOC 사다리·소진모드·큰수매수 세부는 제외 — 원문 미확보 규칙에 창작을 넣지 않는다.
  • 검증 = 다른 종목 dryRun (사용자 확정): TQQQ v2.2 LIVE는 그대로 두고, V4.0은 다른 종목(권장 SOXL)의 신규 dryRun 사이클로 검증. 심볼당 active 1개 제약 유지.
  • 아키텍처 = A안 (사용자 확정): 사이클 테이블 확장(nullable) + 순수 코어 strategy-v4.ts + 공용 체결 동기화 lib sync.ts (worker 편입).

4. 코어 전략 — src/lib/infinite-buy/strategy-v4.ts (순수 함수)

입력 V4State

{ t, splits, cash, starBase, bigBuyPremium,
  avgPrice, currentPrice, holdingQty }   // LIVE=토스 스냅샷 / dryRun=가상 체결 이력에서 파생(§6)

일일 계산

별%       = starBase × (1 − 2t / splits)
별지점    = px(avgPrice × (1 + 별%/100))      // px = 소수 2자리 반올림
매수점    = 별지점 − 0.01
1회매수금 = cash / (splits − t)               // t > splits−1 이면 소진(주문 없음)

주문 규칙 (전부 매일 신규 발주, LOC=LIMIT+CLS, 지정가=LIMIT+DAY)

국면조건매수매도(보유>0 공통)
첫매수t=0 && 보유=0큰수 LOC: 리밋 px(현재가×(1+bigBuyPremium%)), 수량 floor(1회매수금÷리밋)—
전반전0 < t < splits/2½: floor(반회분÷매수점) @매수점 + ½: floor(반회분÷평단) @평단쿼터 floor(N×0.25) @별지점 LOC + 나머지 N−floor(N×0.25) @px(평단×(1+starBase%)) 지정가. 쿼터 0주면 생략
후반전splits/2 ≤ t ≤ splits−1전액: floor(1회매수금÷매수점) @매수점
소진t > splits−1주문 없음 → exhausted 플래그 반환 (호출부가 status 변경)
  • 반올림(§11-5 미확정 → v2.2 관행 채택): 쿼터 floor, 나머지를 지정가에 몰아 매도 합계 = 보유수량. 수량 0 주문은 생략.
  • 출력: orders[] + exhausted 플래그. T·잔금 갱신은 반환하지 않는다 — 상태 갱신은 체결 동기화(§5)만이, 사이클 종료(completed) 판정은 apply 단계(§7-②)만이 수행.

5. 체결 동기화 — src/lib/infinite-buy/sync.ts (신규 공용 lib)

syncFills(cycle)

  • 토스 CLOSED 주문 조회(사이클 생성일 이후·심볼 필터) → tossOrderId 매칭 → BUY·SELL 모두 filledQty/filledPrice/filledAt 기록. 이미 filled·수량 동일이면 skip(멱등).
  • 기존 서버 액션 syncSellFills는 이 lib을 부르는 얇은 래퍼로 교체(대시보드 버튼 동작 유지, BUY까지 반영되도록 확장).

applyFillsToV4State(cycle) — 사용자 승인된 해석 2건 포함

  • stateApplied가 아닌 filled/simulated_filled 주문을 체결시각 순으로 반영:
  • 매수 (해석 ①, 주문 단위 비례): 전반전 두 주문은 각각 절반치 → ΔT = 0.5 × (체결수량÷주문수량). 후반 전액·첫매수 주문 → ΔT = 1 × (체결수량÷주문수량). 전량 체결 시 원문 검증점 +1/+0.5와 일치.
  • 쿼터매도(sell_loc_star): T ← T × 0.75. 지정가매도(sell_lim_target): T ← T × 0.25. 매도 배수는 부분체결이어도 전량 기준 적용(원문에 세분 규칙 부재 — 확정). 같은 날 지정가매도와 매수가 함께 체결되면 ×0.25 먼저, 매수 ΔT 나중(원문 §2 "×0.25 + 1/0.5").
  • 잔금 (해석 ②, 체결 기반): 매수 cash −= 체결가×체결수량, 매도 cash += 체결가×체결수량.
  • 반영 후 stateApplied=true. CLOSED 주문만 동기화하므로 filledQty는 최종값 — boolean 마커로 충분(증분 추적 불필요).
  • 같은 거래일 내 적용 순서(체결시각 동일 케이스): sell_lim_target(장중) → sell_loc_star(종가) → 매수(종가). 쿼터매도와 LOC 매수는 1센트 분리로 같은 날 동시 발동 불가.

정합성 크로스체크 (LIVE 전용, dryRun은 skip)

  • 동기화 후 Σ매수체결 − Σ매도체결 ≠ 토스 실제 보유수량이면: 해당 사이클 그날 주문 보류 + 경고 로그 + 대시보드 배지. 수동 매매 혼입·동기화 누락으로 오염된 T로 주문 내는 것을 막는 마지막 방어선.

실패 정책

  • 동기화 실패(토스 API 에러 등) 시: v4.0은 그날 주문 보류(다음 15분 폴에서 자동 재시도), v1/v2.2는 경고 후 진행(회차가 체결 비의존). — 레퍼런스 §10 to-be 그대로.

6. dryRun 가상 체결 — src/lib/infinite-buy/dry-fill.ts

  • dryRun 사이클은 실제 체결이 없으므로, 다음 실행 시 전일 일봉(getDailyCandles)으로 전일 simulated 주문의 체결을 판정한다:
주문체결 조건체결가
LOC 매수종가 ≤ 리밋종가
LOC 매도(쿼터)종가 ≥ 리밋종가
지정가 매도고가 ≥ 리밋리밋가
  • 판정 로직은 순수 함수(일봉 주입)로 작성해 단위 테스트. 체결 시 simulated_filled + filledQty/filledPrice 기록, 미체결은 expired로 종료(다음날 재판정 방지) → 라이브와 동일한 체결 적용 경로로 T·잔금 갱신. dryRun에서도 평단·보유는 가상 체결 누적으로 추정(라이브 스냅샷 없음): 사이클 자체 필드가 아닌 filled 이력에서 평단·보유를 파생해 V4State를 구성한다.
  • 고가 판정을 위해 DailyCandle(toss/client.ts)에 high 필드 추가(응답에 없으면 close로 대체 — 보수적 판정). 기존 소비자(차트·RSI)는 close만 사용하므로 무영향.

7. 실행 흐름 (worker · run)

infinite-buy-loop (기존 15분 폴, US 정규장 거래일당 1회) — v4 사이클 처리 순서:
① sync      : LIVE → syncFills / dryRun → dry-fill 판정
② apply     : applyFillsToV4State → T·잔금 갱신 (보유 0 && t>0 → completed 종료)
③ check     : LIVE 크로스체크 (불일치 → 그날 보류 + 경고)
④ compute   : computeDailyOrdersV4 (exhausted → status 변경 후 종료)
⑤ submit    : 기존 buying-power 체크 → 주문 제출/로깅 (dryRun은 simulated 기록)
  • v4 실행은 run.ts 수정 대신 신규 run-v4.ts 오케스트레이터가 ①~⑤ 전체를 한 함수로 강제(②를 건너뛴 ④가 구조적으로 불가능). worker와 수동실행 액션(runCycleNow)이 공유. 기존 v1/v2.2 경로(run.ts)는 무변경.
  • 불변식: "상태 갱신은 항상 최신 체결 반영 후" — ②를 건너뛴 ④는 금지.
  • 주문은 거래일당 1회(lastRunDate 관행 유지), sync/apply는 폴마다 실행해도 멱등이라 안전.

8. UI · 액션

  • createCycle: version "v4.0" 옵션. starBase 자동(TQQQ 15/SOXL 20), 그 외 심볼은 입력 필수. tValue=0, cashRemaining=principal 초기화. 항상 dryRun 시작(기존 관행).
  • CycleView/대시보드: v4 사이클에 T(소수 표시)·잔금·오늘의 별지점·1회매수금 표시, exhausted·크로스체크 불일치 경고 배지.

9. 에러 처리 · 가드

  • 기존 assertSubmittable(수량·가격 유효성) 재사용.
  • v4 전용 가드: 잔금 ≤ 0, starBase 미설정, t=0 && 보유>0(사이클 밖 기보유 혼입) → 그날 주문 보류 + 경고.
  • INFINITE_BUY_DISABLED=1 킬스위치·매수가능금액 체크 등 기존 안전장치 그대로 적용.
